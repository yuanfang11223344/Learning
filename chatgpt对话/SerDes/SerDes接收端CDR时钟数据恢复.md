## 摘要

CDR通过接收数据边沿中的相位信息动态调整采样时刻，使采样点尽量保持在眼图中心。它的核心目标不是给所有后级逻辑提供一个通用高速时钟，而是让接收端能在正确的UI位置完成可靠采样和判决。CDR依赖足够的数据跳变，因此编码和加扰常用于保证跳变密度。

## 关键词

SerDes, CDR, Clock Data Recovery, PLL, 采样相位, 眼图中心, 跳变密度, 8b10b, 加扰

## 核心理解

- CDR从数据跳变中提取相位误差信息，经过环路滤波后调整VCO/PI/采样相位。
- 长连续0或1会缺少边沿信息，使CDR无法持续校正相位，因此协议会限制run length或使用加扰。
- 恢复出来的高速采样相位主要服务于接收判决；并行域时钟常由本地PLL或相关时钟体系产生，不一定直接由CDR高速时钟简单分频。
- CDR是动态闭环过程，受信道抖动、均衡状态、数据模式和环路带宽共同影响。

## 可执行学习建议

## 1. CDR 要解决什么问题？

SerDes 接收端收到的是一串高速串行数据，例如：

```text
... 1 0 1 1 0 0 1 0 ...
```

但是接收端本地并不知道这些 bit 的精确边界在哪里。

理想情况下，每个 bit 占用一个 UI：

```text
| bit0 | bit1 | bit2 | bit3 |
```

最佳采样点应该在每个 UI 的中间位置，因为这里距离左右边沿最远，抗抖动能力最好。

问题是：

```text
发送端时钟和接收端时钟不可能完全一样；
信道会引入延迟、抖动、ISI；
接收到的数据边沿会漂移；
接收端不知道该在什么时候采样。
```

所以 CDR 的任务就是：

**从接收数据的跳变边沿中提取时钟相位信息，并不断调整采样时刻，使采样点保持在数据眼图中心。**

---

## 2. CDR 为什么依赖数据跳变？

CDR 不能凭空知道 bit 边界，它只能观察数据什么时候发生跳变。

例如接收数据为：

```text
1 0 1 0 1 0
```

每个 bit 都有跳变，CDR 很容易判断边沿位置。

但如果数据是：

```text
1 1 1 1 1 1 1 1
```

或者：

```text
0 0 0 0 0 0 0 0
```

中间没有跳变，CDR 就没有新的相位参考，只能靠自己原来的时钟继续跑。

如果这段连续 0 或连续 1 太长，接收端采样时钟会因为频率误差和抖动慢慢偏离最佳位置，最后可能采到错误数据。

所以 CDR 要求输入数据必须有足够的跳变密度。

这就是为什么 SerDes 发送端通常要做：

```text
8b/10b 编码
64b/66b 加扰
128b/130b 加扰
其他 scrambling / encoding
```

它们的目的之一就是避免数据长期不翻转，保证 CDR 有足够的边沿可以跟踪。

不过要注意：

**8b/10b 是严格限制连续相同位数量，并且兼顾直流平衡；加扰通常是统计意义上打散数据，并不一定绝对限制最长连续 0/1。**

## 3. CDR 的基本结构

可以把 CDR 看成一个反馈控制系统：

```text
接收串行数据
     │
     ▼
相位检测器 PD
     │
     ▼
环路滤波器 Loop Filter
     │
     ▼
VCO / PLL / Phase Interpolator
     │
     ▼
采样时钟
     │
     └────反馈给采样器，继续检测数据边沿
```

更直观一点：

```text
数据边沿早了？  →  CDR 把采样时钟往前调
数据边沿晚了？  →  CDR 把采样时钟往后调
刚好居中？      →  保持当前采样相位
```

它不是一次性恢复完就结束，而是持续调整。

---

## 4. CDR 如何找到最佳采样点？

以 NRZ 数据为例，一个 UI 内最理想的采样位置在中间：

```text
bit 边界       最佳采样点       bit 边界
   │               │              │
   ▼               ▼              ▼
---+---------------●--------------+---
```

如果采样点太靠左：

```text
---+----●-------------------------+---
       太早
```

那么容易受到前一个 bit 的影响。

如果采样点太靠右：

```text
---+-------------------------●----+---
                           太晚
```

那么容易受到下一个 bit 的影响。

CDR 会利用数据跳变边沿判断采样时钟是“早”还是“晚”。

例如常见的 Bang-Bang CDR / Alexander CDR 会用多个采样点判断：

```text
Early sample    Data sample    Late sample
     │              │              │
     ▼              ▼              ▼
---- x ------------ x ------------ x ----
```

如果检测到边沿更靠近 Early，说明时钟偏晚，需要调整。

如果检测到边沿更靠近 Late，说明时钟偏早，也需要调整。

最终目标是让数据采样点落在眼图中间。

## 5. CDR 恢复的是“高速采样时钟”，但不一定直接给用户逻辑用

你这句话需要稍微区分：

> CDR 恢复出来的时钟并没有直接给其他逻辑或者电路使用。

这个说法在很多 FPGA / SerDes 架构中基本可以这样理解，但要更准确一点：

**CDR 恢复出来的时钟首先用于接收端高速串行采样和串并转换。**

也就是说，它一定会参与 RX PMA 内部的数据采样，例如：

```text
高速串行输入
     │
     ▼
CDR 恢复采样时钟
     │
     ▼
高速采样器
     │
     ▼
串并转换器
```

但是，恢复出来的高速时钟通常不会直接拿给普通 FPGA 逻辑大范围使用。

原因是：

```text
1. 它频率很高，普通逻辑无法直接使用；
2. 它带有从数据中恢复出来的抖动；
3. 它属于接收高速模拟/混合信号部分；
4. 后级 PCS、MAC、用户逻辑通常需要更干净、更稳定、更容易约束的并行时钟；
5. SerDes 内部常通过分频、缓冲、弹性 FIFO、时钟校正等方式隔离时钟域。
```

所以更准确的说法是：

**CDR 时钟主要用于 RX 高速采样和内部恢复；并行用户逻辑使用的 RXUSRCLK、RXOUTCLK、fabric clock 等，可能来自恢复时钟的分频，也可能经过缓冲、PLL/MMCM、弹性缓冲或本地参考时钟体系处理，而不是简单把 CDR 高速时钟直接分频后裸用。**

不同厂商、不同 SerDes 架构会有区别。

## 6. CDR 与本地 PLL 的关系

接收端通常有本地参考时钟，例如：

```text
156.25 MHz
322.265625 MHz
100 MHz
125 MHz
```

这些参考时钟送入 PLL，产生 SerDes 内部所需的高速时钟范围。

但是仅靠本地 PLL 还不够。

因为发送端的数据速率和接收端本地时钟之间一定存在微小偏差，比如：

```text
发送端实际速率：10.3125001 Gbps
接收端本地估计：10.3124998 Gbps
```

两者看起来非常接近，但长时间运行后，采样点会逐渐漂移。

因此 CDR 会在本地 PLL 产生的时钟基础上，根据接收数据边沿不断微调相位，甚至微调等效频率。

可以理解为：

```text
本地 PLL：提供一个大致正确的高速时钟基础
CDR 环路：根据数据跳变不断修正相位和频率
```

## 7. CDR 的工作过程举例

假设一个 UI 是 100 ps，也就是 10 Gbps 左右的数据速率。

理想采样点在 UI 中间：

```text
0 ps        50 ps        100 ps
边沿        采样点        边沿
```

一开始 CDR 采样点可能偏早：

```text
0 ps    35 ps            100 ps
边沿    采样点            边沿
```

CDR 观察到数据跳变后，会判断当前采样相位偏早，于是把采样点往后调。

调整后：

```text
0 ps        45 ps        100 ps
边沿        采样点        边沿
```

继续跟踪：

```text
0 ps        50 ps        100 ps
边沿        采样点        边沿
```

最终采样点靠近眼图中心。

如果信道温度变化、抖动变化、发送端频率微漂移，CDR 又会继续动态调整。

所以 CDR 不是静态模块，而是一个持续工作的反馈环路。

## 8. CDR 为什么怕连续 0 或连续 1？

假设数据为：

```text
1010101010
```

每个 UI 都有跳变，CDR 可以频繁校准。

但如果数据为：

```text
1111111111111111
```

中间没有跳变，CDR 就无法知道采样点是否偏移。

如果接收端时钟略快，采样点会慢慢向前漂移：

```text
初始：采样在 UI 中间
之后：采样点逐渐靠近左边沿
再后：可能采到前一个 bit 或边沿附近
```

如果接收端时钟略慢，采样点会慢慢向后漂移：

```text
初始：采样在 UI 中间
之后：采样点逐渐靠近右边沿
再后：可能采到下一个 bit 或边沿附近
```

因此协议会规定最大连续相同位数，或者用编码/加扰保证跳变密度。

## 9. 8b/10b 对 CDR 的帮助

8b/10b 编码把 8 bit 原始数据编码成 10 bit 传输码。

它主要有几个作用：

```text
1. 限制连续 0 或连续 1 的长度；
2. 增加数据跳变密度，方便 CDR 锁定；
3. 保持直流平衡，避免链路基线漂移；
4. 提供特殊控制字符，例如 K28.5，用于字节对齐。
```

例如原始数据可能是：

```text
00000000
```

如果直接发送，就可能造成很长的连续 0。

经过 8b/10b 编码后，会变成具有一定跳变的 10 bit 码型，CDR 更容易维持锁定。

---

## 10. 加扰对 CDR 的帮助

64b/66b、128b/130b 等高速协议通常不再使用 8b/10b，而是使用加扰。

加扰器会把原始数据打散，使输出数据看起来更随机：

```text
原始数据：
000000000000000000000000

加扰后可能变成：
101101001011010111001010
```

这样可以提高跳变概率，减少长时间无跳变的情况。

但是加扰是统计型方法，不像 8b/10b 那样严格限制每个码组的特性。

因此高速协议还会配合：

```text
同步头
block alignment
scrambler polynomial
训练序列
comma / alignment marker
elastic buffer
```

来共同保证接收端稳定工作。

## 11. CDR 与眼图的关系

CDR 的最终目标可以从眼图理解。

未正确恢复采样点时：

```text
采样点靠近边沿，眼图开口利用不足
```

正确锁定后：

```text
采样点位于眼图中央，误码率最低
```

所以 CDR 实际上在做两件事：

```text
1. 从数据边沿恢复 bit rate 对应的时钟相位；
2. 调整采样点，使采样位置位于眼图最佳区域。
```

如果信道 ISI 很严重，眼图变窄，CDR 的工作也会变困难。

因此 SerDes 接收端通常还会配合：

```text
CTLE
DFE
VGA
CDR
Deserializer
PCS
```

其中：

```text
CTLE / DFE：改善眼图
CDR：找到合适的采样时间
Deserializer：完成串并转换
PCS：完成解码、对齐、纠错等处理
```

## 12. CDR 的锁定过程

CDR 通常有两个阶段：

### 第一阶段：锁频

刚开始时，接收端不知道数据速率的精确频率，只能依赖本地参考时钟和 PLL 先接近目标速率。

这个阶段主要是让本地采样时钟频率接近数据速率。

### 第二阶段：锁相

频率接近后，CDR 根据数据跳变不断调整采样相位。

目标是让采样点落在 UI 中央。

锁定后，CDR 会继续跟踪数据变化，而不是停止工作。

## 13. 总结成一句话

**CDR 是 SerDes 接收端中根据接收数据跳变信息进行相位/频率跟踪的反馈系统，它利用 PLL、相位检测器和环路滤波等结构恢复高速采样时钟，并动态调整采样点，使接收端能够在眼图中心稳定采样数据；为了保证 CDR 能持续锁定，发送端通常需要采用 8b/10b 编码或加扰来提高跳变密度，避免长时间连续 0 或连续 1。**
